## 1. 배경 — 어떤 구조적 문제가 있었나

카피아는 신차구매·중고금융·시승·캐시백 4가지 서비스를 단일 플랫폼에서 운영한다. 마케팅팀은 33개 채널, 296개 UTM 링크를 운영하며 매월 수십만 건의 트래픽을 만들어내고 있었다.

그러나 **채널 성과를 확인하려면 개발팀에 데이터 추출을 요청하거나 수작업 엑셀 정리가 필요했고, 결과를 받기까지 최소 2~3일이 걸렸다.** 그 사이에 예산을 조정할 타이밍을 놓치는 일이 반복됐다.

더 근본적인 문제는 데이터가 없는 게 아니라 **데이터를 읽는 구조가 없다는 것**이었다.

- UTM 그룹이 "신차 프로모션" / "중고차 금융" 두 가지로만 분류되어 있었고, 헤이딜러·엔카 같은 채널 개념이 DB에 존재하지 않았다. 헤이딜러가 보낸 클릭 48만 건이 어느 버튼에서 왔는지 구분이 불가능했다.
- 신차 목적으로 만든 UTM 그룹 안에 중고차 링크가 섞여 있었다. 이 구조에서는 신차 채널이 중고금융 전환을 만들어도 이를 "의도된 전환"인지 "이탈"인지 구분할 방법이 없었다.
- 신차와 중고차 사업부를 한 팀이 담당하는 구조임에도 사업부별 성과를 분리해서 보는 화면이 없었다.

누군가 수작업으로 채널 데이터를 집계하지 않으면, 마케팅 팀은 클릭수 외에 아무것도 볼 수 없었다.


## 2. 내가 정의한 문제 (Problem Framing)

처음에는 "대시보드가 없어서 불편하다"는 요청처럼 보였다. 그런데 실제 어드민의 UTM 그룹 44개와 링크 296개를 전수 분석하자 문제의 본질이 다르게 보였다.

**"대시보드 부재" 문제가 아니라 "집계할 수 없는 데이터 구조" 문제였다.**

대시보드를 만들어도 현재 DB 구조로는 "헤이딜러에서 발생한 신차 전환수"를 쿼리할 방법 자체가 없다. 링크와 채널 사이의 연결고리(channel_id)가 없고, 링크의 목적(purpose)이 정의되어 있지 않기 때문이다.

그래서 문제를 이렇게 재정의했다.

> "채널 → UTM그룹 → 링크의 3단계 계층이 DB에 구축되어야만, 대시보드가 의미있는 숫자를 보여줄 수 있다. 화면보다 구조가 먼저다."

이 정의가 프로젝트 전체의 방향을 결정했다. 대시보드 UI 설계보다 데이터 구조 설계에 더 많은 시간을 투자한 이유다.


## 3. 내가 만든 구조적 변화 (Solution Structure)

### 핵심 설계 결정

**결정 1: 채널 마스터 테이블 신설 (channel_master)**

UTM 그룹 이름에서 채널명을 파싱하는 방식을 고려했지만, 그룹명이 사람마다 다르게 입력되어 있었다("네이버 GFA 광고", "GFA_커뮤니티애즈 벤츠", "네이버GFA" 등). 파싱 규칙에 의존하면 데이터가 늘어날수록 예외 케이스가 누적된다.

대신 channel_master라는 별도 테이블을 만들어 채널을 정규화했다. UTM 그룹이 channel_id FK로 참조하는 구조이므로, 채널명이 어떻게 입력되어도 집계는 항상 channel_id 기준으로 정확하게 이루어진다. 헤이딜러 구버전 그룹(클릭 259,541건)도 현재 그룹(227,410건)도 같은 ch_011로 병합된다.

**결정 2: 섹션 탭 필터 기준 — linkType이 아닌 purpose(B안)**

신차 탭에서 어떤 데이터를 보여줄 것인지에 대해 두 가지 방식을 검토했다.

- A안: 클릭수는 전체, 전환수만 신차 필터 → "신차 링크가 만든 전환"과 "전체 클릭" 사이의 괴리가 생겨 전환율이 왜곡됨
- B안: 클릭수와 전환수 모두 purpose="new-car" 링크만 집계 → 탭 이름과 데이터 기준이 일치. 신차 링크에서 중고금융 전환이 발생한 것(크로스전환)도 측정 가능

B안을 선택했다. 단기적으로는 숫자가 더 작게 보이더라도, 의사결정의 근거가 되는 숫자는 정확해야 한다고 판단했다.

**결정 3: dashboardView 속성으로 모달 표시 방식 분기**

링크 단위 성과가 중요한 채널(헤이딜러: 버튼별 전환율 9.7%~59.1%)과 그룹 합산이 의미있는 채널(엔카: 차량별 링크 26개)이 섞여 있었다. 단일 방식으로 통일하면 한쪽은 반드시 불편해진다.

UTM 그룹 생성 시 마케터가 "링크단위" / "그룹합산"을 선택하고 이를 dashboard_view 컬럼에 저장하는 방식을 택했다. 어드민에서 결정한 값이 대시보드 UX를 자동으로 결정하는 구조다. 마케터가 설정하면 개발팀이 코드를 건드릴 필요가 없다.

### 구현 개요

데이터 레이어 설계 (channel_master DDL, utm_groups 컬럼 추가, 집계 스키마 MONTHLY_BY_LINK / DAILY_BY_LINK) → 채널 체계 정리 (33개 채널, 10개 그룹 분류, 목적별 매핑) → 대시보드 UX 설계 (3-Layer: 채널 테이블 → 드릴다운 모달 → 링크 상세) → React JSX 프로토타입 v1~v5 반복 구현 → 개발팀 전달 문서 작성 (기획서, 기능정의서 v2, Jira Epic/Sub-task 7개).

모든 구현은 **"실제 카피아 채널 데이터(클릭수·전환수 실수치)"를 목업에 반영**하여, 개발팀이 API를 연결하기 전에도 화면이 어떻게 작동할지 정확하게 확인할 수 있도록 했다.

| 지표 | 변화 전 | 변화 후 | 비고 |

| 채널 성과 확인 소요 시간 | 2~3일 (개발팀 요청) | 실시간 (대시보드 접속) | API 연동 완료 시 |

**정성적 변화:**

- 마케팅팀이 개발팀 요청 없이 채널 성과를 스스로 조회하는 구조가 처음으로 만들어졌다.
- 헤이딜러처럼 압도적인 채널이 있어도 "채널 제외 토글"로 나머지 채널의 상대적 성과를 왜곡 없이 비교할 수 있게 됐다.
- 기존에 감지조차 못했던 크로스전환(신차 링크 → 중고금융 전환)이 수면 위로 올라와, 채널별 실제 기여를 더 정확하게 평가할 수 있는 기반이 생겼다.
- 인플루언서 채널(유튜브·인스타그램)이 추가될 때 채널 마스터에 1행만 INSERT하면 대시보드에 자동 반영되는 확장 구조가 설계됐다.


## 5. 이 프로젝트에서 내가 한 판단들

**판단 1: 대시보드 설계보다 데이터 구조 설계를 먼저 했다**

요청은 "대시보드를 만들어달라"였다. 그러나 현재 DB 구조로는 채널 단위 집계 쿼리 자체가 불가능하다는 걸 분석 과정에서 발견했다. 화면을 아무리 잘 만들어도 데이터 구조가 없으면 의미있는 숫자를 보여줄 수 없다. 개발팀 설득을 위해 기획서보다 DDL과 집계 스키마를 먼저 작성했다.

**판단 2: 마케터 선행 작업을 개발 전에 완료하도록 설계했다**

편의점·포람페·동호회 등 7개 채널에 신차/중고 링크가 섞여 있었다. 개발팀이 소급 매핑 작업을 먼저 하면 잘못된 데이터가 고착될 위험이 있었다. 작업 순서를 "마케터 링크 정리 → DB 스키마 변경 → 소급 매핑 → API 구현" 순서로 명시하고 Jira 의존성으로 연결했다.

**판단 3: 헤이딜러 구버전 그룹을 삭제하지 않고 병합했다**

"미사용"으로 표시된 헤이딜러 구버전 그룹(클릭 259,541건)이 실제로는 현재도 유입이 이루어지고 있었다. 삭제하면 과거 데이터가 유실되고 현재 트래픽도 집계에서 빠진다. channel_id ch_011에 병합하고 대시보드에서 함께 집계하는 방식을 택했다.

**판단 4: 섹션 탭 필터를 A안이 아닌 B안으로 결정했다**

A안(클릭 전체, 전환만 필터)이 구현이 단순하고 숫자가 더 크게 보인다. 그러나 "신차 탭 전환율"이 실제 신차 링크 기준으로 계산되지 않으면 그 숫자로 의사결정을 할 수 없다. 단기적으로 숫자가 낮아 보이는 불편함보다, 의사결정 근거로 쓸 수 있는 정확성을 선택했다.

**판단 5: 인플루언서 채널을 콘텐츠 그룹에서 분리했다**

네이버 블로그와 인플루언서 제휴 링크는 성격이 다르다. 블로그는 자사 콘텐츠 자산이고 인플루언서는 외부 개인 크리에이터다. 지금은 규모가 작아 같은 그룹으로 묶어도 불편함이 없지만, 유튜브·인스타그램으로 확장될 경우 플랫폼별 성과 비교가 필요해진다. 확장 시점이 아닌 설계 시점에 분리했다.


## 6. 한계 및 다음 단계

**현재 한계:**

- 대시보드는 프로토타입(React JSX 목업) 단계로, 실제 DB API 연동이 완료되지 않았다. 현재 화면의 수치는 실제 클릭/전환 실수치 기반으로 만든 근사값 목업이다.
- 편의점·포람페 등 링크 혼재 그룹 정리는 마케터 선행 작업이 필요하며 아직 완료되지 않았다.
- 마케팅 비용(CPC) 데이터가 연동되지 않아 채널별 ROI(CPL) 계산은 아직 불가능하다.

**개선하고 싶은 방향:**

- API 연동 완료 후 실수치 기반 대시보드 운영
- 채널별 목표 전환율 설정 + 실적 비교 기능 (마케팅 OKR 연동)
- 네이버 광고·구글 광고 비용 데이터 연동으로 CPL 자동 계산


## 7. 포트폴리오 활용 메모 (내부용 — 외부 노출 금지)

- **강조해야 할 JD 유형:** AI 도구 활용 기획자 / 마케팅 인텔리전스 / 데이터 기반 의사결정 시스템 설계 / PO·기획자 (B2B SaaS 경험) / AX(AI Transformation) 관련 포지션
- **민감 정보 처리:** 회사명(카피아) 노출 가능. 클릭수·전환수는 실수치 공개 가능 (공개된 어드민 화면 기준)
- **이 프로젝트로 말하고 싶은 핵심 한 문장:** "나는 데이터가 있어도 읽을 수 없었던 조직에 데이터를 읽는 구조를 설계했고, AI와 협업하여 개발 리소스 없이 프로덕션 수준의 스펙을 혼자 완성했다."
