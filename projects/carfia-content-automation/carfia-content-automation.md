## 1. 배경 — 어떤 구조적 문제가 있었나

카피아의 콘텐츠 운영은 4가지 카테고리(뉴스 요약 / 프로모션 정보 / 자산 콘텐츠 / 고객 후기)로 나뉘는데, 이 모든 단계를 콘텐츠 담당자 1인이 수동으로 처리하는 구조였다. 특히 뉴스·프로모션은 시의성이 핵심인데도 발행 속도가 정보 갱신 주기를 따라가지 못했다.

자동차 브랜드별 월 프로모션이 매월 1일 일제히 갱신되는데, 담당자가 수동으로 정리하느라 첫 주 동안 사용자는 이미 지난달 정보를 보고 있었다.
뉴스 큐레이션은 원문 복사 시 저작권 리스크, 직접 다시 쓰면 시간 부족 → 발행 자체를 포기하는 날이 빈번했다.
담당자가 부재하거나 작업자가 바뀌면 프로세스 전체가 멈추는 단일 의존 구조였다.

가장 결정적인 신호: 담당자가 "SEO·브랜드 신뢰에 가장 중요한 자산 콘텐츠를 쓸 시간이 없다"고 토로했고, 실제 자산 콘텐츠 비중이 전체의 8%에 불과했다. 단순한 양 부족이 아니라 "양 때문에 질을 못 챙기는" 구조적 문제였다.


## 2. 내가 정의한 문제 (Problem Framing)

처음 받은 요구사항은 "콘텐츠 발행량을 늘리고 싶다"였지만, 이를 그대로 받아들이지 않았다.
재정의한 문제: 발행량 부족은 증상이고, 본질은 **"반복형 콘텐츠가 고가치 콘텐츠의 시간을 잠식하는 구조"**다.
따라서 목표를 "발행량 N% 증가"로 설정하지 않고, **"반복형 콘텐츠를 자동화해 담당자가 자산형 콘텐츠에 집중할 수 있게 만드는 것"**으로 잡았다. 이 재정의 덕분에 단순 양적 KPI가 아니라 카테고리별 포트폴리오 비중 변화를 추적 지표로 가져갈 수 있었다.
비즈니스 관점에서 이 문제는 "속도 문제"가 아니라 **"확장 불가능한 구조 문제"**였다. 사용자가 2배가 되어도, 담당자가 바뀌어도, 동일 품질로 발행이 유지되는 시스템이 필요했다.


## 3. 내가 만든 구조적 변화 (Solution Structure)

### 핵심 설계 결정

- 결정 1: **카테고리별 분리 프롬프트** (통합 프롬프트 대신) — 공통 지침(브랜드 톤·페르소나·금지사항) + 카테고리별 모듈 구조로 분리. 브랜드 일관성과 카테고리별 전환 목표 특화를 동시 확보.
- 결정 2: **자동발행 vs Draft 검수 — 카테고리별 차등 적용** — 뉴스 요약: 출처 검증 시 자동발행 / 프로모션: 무조건 사람 검수 / 자산·후기: Draft → 검수 후 발행. 금융 도메인 수치 신뢰 우선.
- 결정 3: **환각 방지 — 프롬프트가 아닌 파이프라인 레벨 검증** — 출처 URL이 없으면 수치 자체를 출력 못하도록 후처리 검증 단계를 파이프라인에 추가(정규식 + 출처 매칭).

### 구현 개요

원칙은 세 가지였다:
- 공통 지침은 시스템 프롬프트에, 카테고리 차이는 분기 모듈에 — 브랜드 일관성과 카테고리별 특화 동시 확보
- 출력은 항상 정형 JSON 스키마 — 후처리·검증·발행 자동화가 가능하도록 구조 강제
- 신뢰가 의심되는 영역은 사람 게이트로 — 자동화 비율을 양보하는 한이 있어도 잘못된 정보 발행은 차단


## 4. 결과 (Outcomes)

| 지표 | 변화 전 | 변화 후 | 비고 |
|------|---------|---------|------|
| 월 발행량 | 25건 | 102건 | +308% |
| 콘텐츠 1건 작업 시간 | 3시간 | 22분 | 검수 시간만 남음 |
| 자산 콘텐츠 비중 | 8% | 35% | 담당자 시간 회수의 직접 증거 |
| 프로모션 갱신 지연 | 매월 5일 | 1일 당일 | 시의성 확보 |
| 카테고리별 톤 일관성 | 담당자 컨디션 의존 | 시스템 보장 | 정성 |

**정성적 변화:**
- 담당자 부재·교체 시에도 발행 프로세스가 중단되지 않는 구조 확보
- 담당자가 자산 콘텐츠·전략 기획에 시간을 쓸 수 있게 되어, 콘텐츠 포트폴리오 자체가 고도화됨
- "딜러보다 내 편" 브랜드 톤이 카테고리·작업자 무관하게 일관 유지


## 5. 이 프로젝트에서 내가 한 판단들

- 판단 1: 요구사항을 그대로 받지 않고 문제를 재정의. "발행량 증가"가 아니라 "반복형이 고가치를 잠식하는 구조" 문제로 재정의.
- 판단 2: 자동화 비율 vs 신뢰 트레이드오프에서 신뢰를 택했다. 프로모션은 자동발행하지 않고 사람 검수 강제.
- 판단 3: 환각 방지를 프롬프트가 아닌 파이프라인 레벨에서 풀었다. 출처 매칭 검증을 후처리 단계로 분리.
- 판단 4: 통합 프롬프트가 아닌 모듈형 구조를 택했다. 카테고리별 전환 목표가 달라 톤·CTA가 뭉개지면 안 된다고 판단.
- 판단 5: 완전 자동발행은 일부러 만들지 않았다. 사람 게이트를 "신뢰 보증 장치"로 설계에 포함.
- 판단 6: 출시 후 프롬프트를 코드처럼 버저닝·수정했다. AI 시스템은 "배포 = 시작"이라는 관점.


## 6. 한계 및 다음 단계

**현재 한계:**
- 프로모션 카테고리는 여전히 사람 검수 의존도가 높음
- 품질 개선이 "주간 스팟체크 → 프롬프트 수정"이라는 수동 루프에 의존

**개선하고 싶은 방향:**
- 카테고리별 평가 셋(eval set)을 만들어 프롬프트 변경 시 회귀 테스트 자동화
- 프로모션 데이터 소스를 정형 입력으로 전환 → 검수 게이트를 줄이고 자동발행 비율 확대
- 사용자 행동 데이터(체류·전환)를 프롬프트 개선 루프에 자동 반영하는 폐쇄형 피드백 루프
