## 1. 배경 — 어떤 구조적 문제가 있었나

헤이딜러(중고차 플랫폼)와 카피아(중고차 금융 중개사)는 이미 비즈니스 협력 구조가 있었지만, 담당자가 카피아 어드민과 헤이딜러 내부 시스템을 번갈아 열람하며 수작업으로 상태를 취합하지 않으면, 동일 고객에 대한 중복 연락·이중 처리·정보 누락이 반복적으로 발생하는 구조였다.

두 시스템 간 실시간 데이터 동기화 없이 운영되었기 때문에, 카피아에서 "서류수신" 이벤트가 발생해도 헤이딜러 담당자는 이를 즉시 알 수 없었다. 금융 조회 결과·서류 처리·약정 상태·정산 정보가 최소 3개 화면에 분산되어 있어, 전체 진행 현황 파악에만 건당 3~5분의 컨텍스트 전환 비용이 발생했다.

또한 양측이 각자 시스템을 개발하면서 API 상태값(status string)의 명칭·공백·대소문자 불일치를 인식하지 못한 채 연동이 진행되고 있었고, 이는 출시 시점에 대규모 운영 장애로 이어질 수 있는 잠재 리스크였다.


## 2. 내가 정의한 문제 (Problem Framing)

표면적으로는 "시스템 간 화면 이동이 불편하다"는 UX 문제처럼 보였지만, 실제로는 "처리량이 늘어날수록 오류율도 함께 늘어나는, 확장 불가능한 운영 구조" 문제였다.

담당자 수를 늘려도 각자가 동일한 수작업 프로세스를 반복할 뿐이었고, 시스템이 상태를 보장하지 않으니 사람이 직접 매번 확인하는 방식이 고착화되어 있었다. 핵심 재정의는 다음과 같다:

> "속도 문제가 아니라 '상태 신뢰성' 문제다. 사람이 상태를 확인하지 않아도 시스템이 정확한 상태를 보장할 수 있어야 한다."

이 관점에서 단순 뷰어(두 시스템 데이터를 한 화면에 보여주는 것)가 아니라, 상태 변경·서류 처리·정산 확인까지 액션이 가능한 운영 도구로 기획 방향을 전환했다.


## 3. 내가 만든 구조적 변화 (Solution Structure)

### 핵심 설계 결정

**결정 1 — 상태값 매핑 테이블을 개발 착수 전에 먼저 완성했다**

양측 코드베이스를 직접 열람하여 카피아의 22종 상태값과 헤이딜러의 Stage1~4Status 타입을 1:1 대조하는 매핑 테이블을 선제적으로 작성했다. 개발 완료 후 통합 테스트로 발견하는 방식 대신 설계 단계에서 불일치를 차단하는 구조를 선택한 이유는, 연동 오류는 수정 비용이 출시 후로 갈수록 기하급수적으로 커지기 때문이다.

**결정 2 — "평균 처리 기간" 지표를 통계 대시보드에서 의도적으로 제외했다**

초기 기획에는 포함되어 있었으나, 이 지표는 헤이딜러 입장에서는 효율 측정 도구지만 공급사인 카피아 관점에서는 "처리 지연의 책임 추궁"으로 읽힐 수 있었다. B2B 파트너십 구조에서 도구가 관계에 부정적으로 작용할 리스크를 판단하여 제외했다. 대신 접수 건수·전환율·단계별 현황 중심의 "성과 확인형" 지표 구조로 재편했다.

**결정 3 — processStage를 단일 소스(customers.ts)로 관리하는 원칙을 수립했다**

코드 분석 결과 process-board.tsx와 execution-requests-page.tsx가 서로 다른 키값을 사용하고 있어 모든 카드 카운트가 0이 되는 Critical 이슈가 존재했다. 이를 해결하는 방법으로 "각 컴포넌트에서 개별 수정"이 아닌 "타입 정의를 단일 소스로 통합하고 모든 컴포넌트가 이를 import"하는 구조적 해법을 제시했다. 단기 수정이 아닌 재발 방지 구조를 목표로 했기 때문이다.

### 구현 개요

전체 구현의 원칙은 **"상태는 시스템이 보장하고, 사람은 판단만 한다"**였다.

카피아 → 헤이딜러 방향의 단방향 상태 동기화를 기본 원칙으로 설정하고, 4단계 금융 프로세스(조회 요청 → 조회완료 → 금융실행 → 종료)를 하나의 ProcessBoard 컴포넌트로 통합했다. 상태별 컬러 배지 체계(Critical/Warning/Info/Success), 고객 상세 모달 정보 구조, 서류 업로드 플로우를 설계했으며, 코드 분석으로 식별한 19건 이슈에 대해 Phase 1(당일·단독 처리) / Phase 2(단기·협의 필요) / Phase 3(방향 결정 후)의 3단계 실행 로드맵을 수립했다.

통계 대시보드는 Chart.js 기반으로 직접 구현하여 HTML 단일 파일로 개발팀에 전달했다.

**정성적 변화:**

- 상태값 혼재(4종 약정 상태, 3종 철회 상태)로 인한 TypeScript 타입 에러가 출시 전 해소됨

- B2B 파트너십 관점에서 공급사(카피아)에게 불리하게 작용할 수 있는 지표를 사전에 걸러내는 기획 기준이 수립됨


## 5. 이 프로젝트에서 내가 한 판단들

**판단 1 — 개발 완료 후 QA가 아니라 설계 단계에서 코드를 직접 열람하는 방식을 선택했다**

통상적인 QA는 기능 완성 후 테스트 시나리오 기반으로 진행된다. 하지만 API 연동 이슈는 양측 코드에서 동시에 발생하는 문제이므로, 런타임에서 발견하면 책임 소재가 불명확해지고 수정 범위도 커진다. 두 파일을 직접 대조하는 "코드 레벨 QA"를 선제적으로 수행함으로써 Critical 7건을 출시 전에 차단했다.

**판단 2 — "그외" 유입처를 헤이딜러로 송출하지 않는 차단 로직의 부재를 Critical이 아닌 C07로 분리하여 별도 처리했다**

이 이슈는 운영 즉시 장애를 일으키진 않지만, 시간이 지날수록 헤이딜러 DB에 불필요한 데이터가 누적되어 통계 정합성을 훼손한다. 당장의 긴급도보다 장기 데이터 품질 리스크로 판단하여, Phase 2 단기 개선 항목으로 분류하고 카피아 API 호출 레이어에서 필터링하는 방향을 제시했다.

**판단 3 — 통계 대시보드에 담당자 개인별 처리 랭킹은 넣되, 처리 속도·응답 시간 지표는 포함하지 않기로 했다**

담당자 랭킹은 동기부여와 성과 확인 목적으로 허용 가능하지만, 처리 속도 지표는 개인 평가 압박으로 작용하거나 카피아와의 파트너십에서 SLA 위반 논쟁의 근거가 될 수 있다. 도구가 내부 갈등을 유발하는 구조를 사전에 차단하는 것이 더 중요하다고 판단했다.

**판단 4 — 해결 제안서를 단순 이슈 목록이 아니라 "Phase별 실행 로드맵" 형태로 구성했다**

이슈를 나열만 하면 개발팀 입장에서는 우선순위 판단 비용이 추가로 발생한다. 외부 협의 없이 당일 처리 가능한 것(Phase 1), 카피아 팀 확인이 필요한 것(Phase 2), 방향 결정이 선행되어야 하는 것(Phase 3)으로 분리함으로써 기획 산출물이 실행 도구가 되도록 설계했다.


## 6. 한계 및 다음 단계

**현재 한계:**

- 통계 대시보드는 목 데이터(Mock) 기반으로 구현되어 있어 실제 API 연동 전 단계임. 실데이터 연결 시 집계 로직 검증 필요
- 카피아 어드민 측 isTerminated 플래그 및 종료 처리 로직은 협의 후 구현 예정으로, 현재 단방향 상태 동기화만 완성된 상태
- QA 분석이 소스코드 정적 분석 기반이므로, 런타임 환경(네트워크 지연, 동시 요청 처리 등) 이슈는 별도 검증 필요

**개선하고 싶은 방향:**

- 실시간 알림 시스템 연동 (부재중 3일 경과 시 Slack 에스컬레이션 자동화)
- 금융사별 승인율 누적 데이터를 기반으로 한 최적 금융사 자동 추천 로직
- 멀티테넌트 구조 전환을 통해 복수 딜러사 운영 가능한 SaaS 플랫폼화
