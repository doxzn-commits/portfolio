## 1. 배경 — 어떤 구조적 문제가 있었나

체험단 활동을 직접 하면서 발견한 문제. 레뷰·강남맛집·미블 등 플랫폼마다 선정 알림과 가이드라인이 분산돼 있고, 체험 전후로 참고해야 할 정보(방문 시간, 촬영 항목, 해시태그, 리뷰 마감일)가 많지만 통합 관리할 수 있는 도구가 없었다.

특히 두 가지가 가장 큰 부담이었다. 첫째, 모든 체험단 내역을 노션에 수기로 입력하는 것. 둘째, 리뷰 작성에 매번 1~2시간이 소요되는 것. 월 3~10개 체험단을 소화하는 크리에이터에게 이 구조는 확장 불가능했다.

**타겟 사용자:** 블로그·인스타그램을 운영하며 월 3~10개 체험단 활동을 하는 24~34세 여성 크리에이터. 여러 플랫폼을 동시에 운영하고 마감 기한이 중요한 프리랜서형 사용자.

**AS-IS:**
- 노션에서 체험단 내역 수기 기록
- 플랫폼마다 따로 접속해 선정 여부 확인
- 마감일 직접 입력, 리뷰는 매번 처음부터 작성

**TO-BE:**
- 선정 공고 URL 하나 붙여넣으면 체험명·마감일·가이드라인 30초 자동 완성
- 체험 당일 아침 브리핑 푸시 + 마감 D-3 알림으로 일정 누락 제로
- AI가 리뷰 초안을 30초 만에 생성 → 리뷰 작성 시간 90% 단축


## 2. 내가 정의한 문제 (Problem Framing)

최초 의도는 "체험단 일정 관리의 분산을 집약화하고, 관리 시간을 극단적으로 줄이는 것"이었다. 그런데 핵심 기능인 URL 파싱이 막히면서 방향이 흔들렸다.

이 시점에서 앱을 그만둘지 고민했지만, 다시 본질로 돌아갔다. **"관리 시간을 줄이겠다"는 목표의 파생에는, 결국 체험단 유저들이 콘텐츠 발행을 더 용이하게 하는 것이 있었다.** 일정 관리는 수단이었고, 목적은 콘텐츠 발행 부담 해소였다.

이 재정의를 통해 세세한 기능 개선과 함께, AI 블로그 콘텐츠 작성 서비스를 핵심으로 한 모델로 방향을 전환해 출시하기로 했다.


## 3. 내가 만든 구조적 변화 (Solution Structure)

### 핵심 설계 결정

- **결정 1 — 기술 스택 (Next.js vs 네이티브 앱):** 빠른 베타 출시를 최우선으로 두고 Next.js를 선택. 이후 Next.js → 앱 형태 리패키징 경로를 확인해 현재 리패키징 작업 예정. 실행 속도를 플랫폼 완성도보다 우선한 판단이었다.

- **결정 2 — 타겟을 2030으로 좁히기:** 완전하지 않은 서비스임을 인정하고, 피드백이 가장 빠르고 직접적인 2030 여성 크리에이터로 타겟을 좁혔다. 이탈률을 감수하더라도 빠른 피드백 수용이 낫다고 판단.

- **결정 3 — 스레드 병행 운영:** 앱 출시 전부터 스레드에서 타겟 유저와 라포를 형성하는 방식으로 커뮤니티와 앱 개발을 병행. 론칭 시 초기 사용자 확보 채널을 미리 만들어두는 구조.

- **결정 4 — URL 파싱 한계 시 방향 전환:** 핵심 기능 구현이 막혔을 때 포기가 아닌 본질 재정의로 대응. "일정 관리"에서 "콘텐츠 발행 용이화"로 핵심 가치를 전환하고 AI 초안 생성을 중심 기능으로 재구성.

### 구현 개요 (현재 진행도)

```
✅ PHASE 0  기반 세팅        — 디자인 시스템, Next.js, Supabase, Vercel
✅ PHASE 1  인증 & 등록 코어  — 소셜 로그인(카카오·구글·애플), URL 파싱 자동 완성
✅ PHASE 2  캘린더 & 상세    — 월간·주간 캘린더, 상세 시트, 6단계 상태 자동 전이(Cron)
──────────────────────── ← 현재
🔜 PHASE 3  매니저 알림      — 웹 푸시, 아침 브리핑, D-3 마감 알림
🔜 PHASE 4  AI 매니저        — AI 리뷰 초안, 사전 알림 이메일 수집
🔜 PHASE 5  탐색 & 수익      — 통합 탐색, 지도, 수익 대시보드
🔜 PHASE 6  바이럴 & 런칭    — 공유 카드, PWA, 온보딩, 론칭
```

## 5. 이 프로젝트에서 내가 한 판단들

- **판단 1 — 핵심 기능 막혔을 때 포기 대신 본질 재정의:** URL 파싱 구현 한계에 부딪혔을 때 앱 중단을 고려했지만, 재정의를 통해 "일정 관리 집약화"에서 "콘텐츠 발행 용이화"로 핵심 가치를 전환. 기능이 아닌 목적 중심으로 방향을 잡았다.

- **판단 2 — 완전하지 않은 서비스를 인정하고 빠른 출시 우선:** 1인 개발 체제에서 완성도를 추구하기보다 빠른 출시 + 피드백 수용이 낫다고 판단. 이탈률 리스크를 감수하고 베타 형태로 먼저 내보내는 방향을 선택.

- **판단 3 — 타겟을 2030 여성 크리에이터로 좁힌 이유:** 반응이 가장 빠르고 직접적인 그룹. 넓은 타겟보다 좁고 뚜렷한 유저에게 먼저 검증받는 것이 제품 개선 속도를 높인다고 판단.

- **판단 4 — 스레드를 앱 개발과 병행한 이유:** 론칭 당일 사용자가 없는 상황을 방지하기 위해, 개발 중에 커뮤니티 채널을 먼저 만들어뒀다. 제품과 유저 채널을 동시에 키우는 구조.

- **판단 5 — Next.js 선택:** 앱 개발 언어와의 고민이 있었지만 실행 속도를 최우선으로 두고 Next.js 선택. 이후 앱 리패키징 경로를 발견해 초기 판단이 결국 효율적인 경로였음을 확인.
