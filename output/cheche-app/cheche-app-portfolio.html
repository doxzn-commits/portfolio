Problem 구조적 문제 3가지
일정·콘텐츠·기록 — 세 가지 모두 수작업 무한반복 구조였다
01
분산된 알림
플랫폼마다 따로 확인하는 선정 알림
레뷰·강남맛집·미블 등 플랫폼별로 로그인해 선정 여부를 직접 확인. 통합된 알림 채널 없음.
실제 영향
선정 놓침 → 체험 기회 상실
D-day 알림 시스템 = 0개
현재 해결책
노션 + 개인 메모 수기 기록
→ 통합 선정 알림 + 자동 일정 등록
02
리뷰 부담
매 체험마다 1~2시간 리뷰 작성 반복
해시태그·필수 촬영 항목·가이드라인을 매번 새로 확인하고, 리뷰 문장을 처음부터 작성. 스케일 불가능.
실제 영향
월 10개 활동 시 리뷰에만
○○시간 이상 소요 (수치 미정)
목표
AI 초안 생성 → 30초 → 수정만
→ AI 리뷰 초안 자동 생성 (90% 단축)
03
수기 반복
모든 체험단 내역을 노션에 수기 입력
선정 공고 URL → 체험명, 마감일, 가이드라인을 직접 복사·붙여넣기. URL 하나에서 30분 이상 소요.
실제 영향
등록 피로 → 관리 포기
확장 불가능한 수작업 구조
목표
URL 붙여넣기 → 30초 자동 완성
→ URL 파싱 자동 등록
Problem Reframing
처음에는 "체험단 일정 관리의 분산을 집약화"하는 요청이었다.
일정 관리 문제가 아니라
콘텐츠 발행 부담 문제였다

플랫폼 분산 집약화

→ URL 파싱 구현 한계로 막힘
→

일정 관리는 수단이었고, 목적은 콘텐츠 발행 부담 해소였다."

기능이 막혔을 때도, 타겟을 좁혔을 때도 — 판단의 이유가 있었다
01
핵심 기능 막혔을 때 — 포기 대신 본질 재정의
URL 파싱 한계 → "일정 집약화"에서 "콘텐츠 발행 용이화"로 핵심 가치 전환. 기능이 아닌 목적 중심으로 재설계.
02
완전하지 않은 서비스를 인정하고 빠른 출시 우선
1인 개발 체제에서 완성도 추구보다 빠른 출시 + 피드백 수용이 낫다고 판단. 베타 형태로 먼저 검증.
트레이드오프: 초기 이탈률 리스크 감수
03
타겟을 2030 여성 크리에이터로 좁힌 이유
반응이 가장 빠르고 직접적인 그룹. 넓은 타겟보다 좁고 뚜렷한 유저에게 먼저 검증받아 제품 개선 속도 극대화.
04
스레드를 앱 개발과 병행한 이유
론칭 당일 사용자 없는 상황 방지. 개발 중에 커뮤니티 채널을 미리 구축 — 제품과 유저 채널을 동시에 성장시키는 구조.
05
Next.js 선택 — 실행 속도 > 플랫폼 완성도
앱 개발 언어 고민했지만 실행 속도 최우선으로 Next.js 선택. 이후 앱 리패키징 경로 확인 → 초기 판단이 효율적 경로였음 확인.
트레이드오프: 초기 UX가 네이티브 앱 대비 제한적
