확장 한계
사용자 2배 증가해도 담당자 없이는 발행 불가
→ 시스템 기반 발행 구조 설계
문제의 본질은 사람에 의존하는 발행 구조였다 — 도구가 아닌 시스템이 필요했다
PROBLEM REFRAMING
처음 받은 요청은 "콘텐츠 발행량을 늘리고 싶다"였다.
발행량이 부족한 게 아니라
반복형이 고가치를 잠식하는
구조 문제였다
A안 · 기존 인식
발행량 N% 증가
양적 KPI 목표
→ 달성해도 구조 문제 미해결
→
B안 · 재정의
반복형을 자동화해 담당자가
자산형에 집중하게
카테고리별 비중 변화가 추적 지표
→ 구조 자체를 바꾸는 설계
이 재정의 덕분에 단순 양적 KPI가 아닌 카테고리별 포트폴리오 비중 변화를 추적 지표로 가져갈 수 있었다.
"사용자가 2배가 되어도, 담당자가 바뀌어도, 동일 품질로 발행이 유지되는 시스템"이 목표였다.
Solution 핵심 설계 결정 3가지
자동화 비율이 아닌 신뢰 구조를 먼저 설계했다
결정 01 · 프롬프트 구조
카테고리별 분리 프롬프트
(통합 프롬프트 대신)
공통 지침(브랜드 톤·페르소나·금지사항) + 카테고리별 모듈 구조로 분리.
고려한 대안
통합 단일 프롬프트
→ 카테고리별 톤·CTA가 뭉개짐
선택한 방식
모듈형 분리 → 브랜드 일관성 + 카테고리 특화 동시 확보
→ 카테고리별 전환 목표 개별 최적화
결정 02 · 자동화 범위
카테고리별 차등 자동화
(일괄 자동화 대신)
금융 도메인 수치 신뢰를 최우선으로 카테고리마다 게이트를 다르게 설계.
프로모션
무조건 사람 검수 — 수치 오류 시 신뢰 훼손
뉴스 요약
출처 검증 완료 시 자동발행
트레이드오프: 자동화율 일부 희생 → 신뢰 확보
결정 03 · 환각 방지
파이프라인 레벨 검증
(프롬프트 지시 대신)
출처 URL 없으면 수치 자체를 출력 못하도록 후처리 검증 단계를 파이프라인에 추가.
구현 방식
정규식 + 출처 URL 매칭 → 검증 실패 시 Draft 전환
원칙
"프롬프트로 설득하지 않는다 — 구조로 막는다"
→ 환각에 의한 오정보 발행 차단
출력은 항상 정형 JSON 스키마 · 신뢰가 의심되면 사람 게이트 · 프롬프트는 코드처럼 버저닝
Data Architecture AI 파이프라인 구조
